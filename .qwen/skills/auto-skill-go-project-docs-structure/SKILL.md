# Go 项目标准文档组织模式

适用于 Go 项目的文档结构化——从零搭建或重构时建立清晰的文档体系，让项目维护者可快速理解开发规则、构建流程、使用方法和踩坑记录。

## 文档结构

```
docs/
├── 01_开发规则.md            # 开发铁律和规范（一次性设定，低频修改）
├── 02_发布与构建流程.md       # 构建、CI/CD、发布流程（随工具链更新）
├── 03_使用说明_YYYYMMDD.md   # 用户使用手册（随功能更新）
└── 04_采坑记录_YYYYMMDD.md   # 踩坑日志（不断追加）
```

### 命名规范

| 规则 | 说明 |
|------|------|
| 前缀 | `序号_` 保证目录排序（01/02/03/04） |
| 文件名 | 中文描述，**不含空格** |
| 日期后缀 | 仅 `03_使用说明` 和 `04_采坑记录` 需要（`_YYYYMMDD.md`） |
| 分隔符 | 使用下划线 `_`，不使用空格或连字符 |

### 为什么 01/02 不需要日期后缀

- `01_开发规则.md` — 项目铁律，变更频率极低，一旦设定很少修改
- `02_发布与构建流程.md` — 随工具链更新，但每次更新时直接覆盖文件，无需保留历史版本
- `03_使用说明_日期.md` — 随功能迭代更新，日期后缀保留版本历史
- `04_采坑记录_日期.md` — 持续追加，日期后缀标记文档版本

## 各文档内容模板

### 01_开发规则.md

项目参与者的"宪法"——不可违背的开发准则。

```markdown
# 项目开发规则

## 核心铁律（必须遵守）

### 1. Git 提交规范
- 每次代码变更务必提交到 Git 仓库
- 所有 commit message 统一使用中文

### 2. 文档同步
- 代码修改必须同步更新 docs/ 目录下的对应文档
- 文档命名：`序号_中文文件名_日期.md`
- 采坑记录模板：背景 → 原因 → 排查过程 → 解决方案 → 建议

### 3. 架构原则
- **小而美**：不做大而全的系统，做瑞士军刀，不过度设计
- **全局视野**：每次改动前必须先理解全局，不能打补丁
- **闭环思维**：所有功能必须形成完整闭环，并有单元测试覆盖

### 4. 技术选型
- **不重复造轮子**：能用行业标准库/框架就用
- **Go 标准**：遵循 Go 官方最佳实践和代码规范

### 5. 代码质量
- 核心代码必须有详细的中文注释（解释为什么这么做的设计意图）
- 单元测试必须写正式脚本，不得写了删删了写
```

### 02_发布与构建流程.md

面向维护者的技术文档——如何构建、测试、发布。

```markdown
# 发布与构建流程

## 一、构建方式

### 快速构建（当前平台）
```bash
./build.sh    # macOS/Linux
.\build.ps1   # Windows
go build ./cmd/<name>/
```

### 跨平台构建（Makefile）
```bash
make build-all    # 构建所有平台到 dist/
make release      # 构建 + 打包压缩包
make clean        # 清理构建产物
```

### 支持平台
| 平台 | GOOS/GOARCH |
|------|-------------|
| macOS Intel | darwin/amd64 |
| macOS Apple Silicon | darwin/arm64 |
| Linux x86_64 | linux/amd64 |
| Linux ARM64 | linux/arm64 |
| Windows x86_64 | windows/amd64 |

## 二、GitHub Actions 自动构建

### 工作流文件
`.github/workflows/build.yml` 包含两个 Job：

**check** — push/PR 到 main：
1. `actions/checkout@v4`
2. `actions/setup-go@v5`（版本从 `go.mod` 自动读取）
3. `go vet ./...`
4. `go build ./cmd/<name>`

**release** — 推送 `v*` 标签：
1. 拉取代码 + 安装 Go
2. `make release`
3. 创建 GitHub Release 并上传 artifacts

### 触发发布
```bash
git tag v1.0.0
git push origin v1.0.0
```

## 三、采坑记录

### 坑1: go.sum 未提交导致 CI 失败
**现象**: CI 中 `go vet` 报 `missing go.sum entry`
**原因**: `go.sum` 未提交到 git 仓库
**解决**: `git add go.sum`

### 坑2: go.mod 缺少 indirect 依赖
**现象**: CI 中 `go vet` 报 `updates to go.mod needed`
**原因**: 初始 `go.mod` 只列了直接依赖
**解决**: `go mod tidy && git add go.mod go.sum`

### 坑3: 某文件缺少 import
**现象**: CI 中 `undefined: X`
**原因**: 本地修改未提交，git 仓库中的版本不完整
**解决**: 对比本地和仓库版本的差异，提交完整文件
```

### 03_使用说明_YYYYMMDD.md

面向用户的文档——如何安装、配置和使用。

```markdown
# 使用说明

> 版本: v0.1.0 | 更新: 2026-07-01

## 一、快速开始

### 下载与启动
```bash
# 从 Release 页面下载对应平台的二进制
./<name>     # macOS/Linux
<name>.exe   # Windows

# 或自行编译
go build ./cmd/<name>/ && ./<name>
```

启动后打开浏览器访问: **http://localhost:8912**
端口可通过环境变量 `PORT` 修改。

### 使用流程
1. 配置管理 → 选择预设或手动填写 → 保存
2. 连接测试 → 选择配置 → 测试连接
3. 聊天测试 → 选择配置 → 输入消息 → 发送
4. 批量测试 → 选择多个配置 → 运行
5. 基准测试 → 设置参数 → 运行（获取 P50/P95 延迟统计）
6. 压力测试 → 设置轮次 → 运行（追踪 Token 消耗）

## 二、配置字段说明
| 字段 | 必填 | 说明 |
|------|:----:|------|
| 配置名称 | ✅ | 唯一标识 |
| API 类型 | ✅ | openai / ollama / azure / custom |
| API 地址 | ✅ | Base URL |
| API Key | ✅ | 支持显示/隐藏切换 |
| 模型名称 | ✅ | 如 gpt-4o-mini |
| 超时(秒) | 可选 | 0=默认 60s |
| 代理地址 | 可选 | 如 http://127.0.0.1:7890 |

## 三、常见问题
### Q: 连接测试成功但聊天测试超时？
**原因**: API 返回了 SSE 流式响应，`io.ReadAll` 永久阻塞直到超时。
**解决**: 确保请求体包含 `"stream": false`。

### Q: 配置存储在哪里？
`~/.<name>/configs.json`，仅当前用户可读写（权限 0600）。
```

## 维护策略

| 文档 | 更新频率 | 更新方式 |
|------|---------|---------|
| 01_开发规则.md | 极少 | 直接覆盖 |
| 02_发布与构建流程.md | 低（工具链变更时） | 直接覆盖 |
| 03_使用说明_日期.md | 中（功能迭代时） | 新建文件 + 旧文件保留 |
| 04_采坑记录_日期.md | 高（每次踩坑后） | 持续追加到同一文件 |

### 采坑记录写作原则

每条记录必须包含以下 5 个要素：

1. **背景** — 什么场景下遇到的（用户操作、CI 运行、代码审查）
2. **根因** — 技术层面的根本原因（不要只写表面现象）
3. **排查过程** — 如何一步步定位到根因（供后来者参考排查思路）
4. **解决方案** — 可执行的修复步骤（代码/命令）
5. **预防建议** — 如何避免再次发生（代码审查清单、自动化检查）

### 命名约定理由

- 序号前缀（01/02/03/04）保证目录排序稳定
- 中文文件名直观可读
- 日期后缀仅用于会频繁更新的文档（说明/采坑），方便追溯版本
- 文件名不含空格，避免跨平台路径问题
