# 顶层架构方案：Probe 测试框架

> 日期: 2026-07-02
> 设计目标: 用统一的 Probe 框架替代零散的 handler 打补丁模式
> 核心思想: 一次架构设计，支撑所有未来测试类型

---

## 一、当前架构的问题（为什么不能再打补丁了）

```
当前:
  handleChat      → 手动构造请求 → 手动解析响应
  handleBatch     → 手动构造请求 → 手动解析响应 → 自行并发
  handleBenchmark → 手动构造请求 → 手动解析响应 → RunBenchmark
  handleBurn      → 手动构造请求 → 手动解析响应 → 自行循环
```

**每个 handler 都在重复做三件事：**
1. 构造 HTTP 请求体
2. 调用 Provider.Chat()
3. 解析响应、计算指标

**加一个新的"探针"（如 json_mode 测试）就要新建一个 handler + 一条路由 + 一个前端标签页。** 这就是"局部无止境打补丁"的根源——架构没有抽象出"测试"这个概念。

## 二、Probe 框架设计

### 2.1 核心接口

```go
// probe/framework.go

// Result 统一测试结果（一个配置 × 一个探针）
type Result struct {
    ProbeName  string             `json:"probe_name"`   // 探针名称
    ConfigName string             `json:"config_name"`  // 配置名称
    Success    bool               `json:"success"`      // 是否成功
    Score      float64            `json:"score"`        // 0-100, -1=不可用
    LatencyMs  float64            `json:"latency_ms"`   // 平均延迟
    Metrics    map[string]float64 `json:"metrics"`      // 自定义指标
    Summary    string             `json:"summary"`      // 一句话结论
    RawData    interface{}        `json:"raw,omitempty"` // 原始数据（用于前端展示）
    Error      string             `json:"error,omitempty"`
}

// Probe 测试探针接口
// 实现此接口即可添加新的测试类型，无需修改框架代码
type Probe interface {
    Name() string                            // 唯一标识
    Description() string                     // 人类可读描述
    Run(ctx context.Context, provider llm.Provider, cfg Config) *Result
}
```

### 2.2 探针配置与套件

```go
// probe/framework.go

// Config 探针执行参数
type Config struct {
    Model     string
    MaxTokens int
    Prompt    string                 // 测试提示词
    Params    map[string]interface{} // 探针特定参数
}

// SuiteConfig 一个待测配置
type SuiteConfig struct {
    Name     string
    Provider llm.Provider
    Config   Config
}

// Suite 测试套件 = 多个配置 × 多个探针
// 自动管理并发执行和结果汇总
type Suite struct {
    Targets []SuiteConfig
    Probes  []Probe
}

// SuiteResult 套件执行结果
// Results 按 [target][probe] 组织
type SuiteResult struct {
    // 每个配置 × 每个探针的结果
    Items []*Result
    // 自动计算的对比摘要
    Comparison *Comparison `json:"comparison,omitempty"`
}

// Comparison 对比分析
type Comparison struct {
    // 每个指标的最佳配置
    BestOf map[string]string `json:"best_of"`
    // 差异百分比
    DiffPct map[string]float64 `json:"diff_pct"`
}
```

### 2.3 内置探针（首批实现）

| 探针 | Name | 测试内容 | 复用现有代码 |
|------|------|---------|:----------:|
| 基准测试 | `benchmark` | 并发多轮测速 + P50/P95 | 复用 RunBenchmark |
| 多轮对话 | `multi_turn` | 多轮对话记忆一致性 | 复用 Chat |
| JSON 模式 | `json_mode` | JSON 结构输出可靠性 | 复用 Chat |
| 行为验证 | `behavior` | 确定性指令服从 | 复用 Chat |
| 温度稳定性 | `temperature_zero` | 低温请求回答一致性 | 复用 Chat |

这些探针都在复用 `llm.Provider.Chat()`，不需要在 Provider 层做任何修改。

## 三、API 层变化

### 新增 1 个端点，替代 3 个端点

```
新增:
  POST /api/test/suite
    请求: { targets: [{name, config}, ...], probes: ["benchmark", "json_mode"], params: {...} }
    响应: { items: [...], comparison: {...} }

保留（前端独立功能，不属探针体系）:
  POST /api/test/chat       → 聊天测试（交互式）
  POST /api/test/connection → 连接测试
  POST /api/test/models     → 模型列表
  POST /api/test/chat/stream → 流式聊天

替代（由 /suite 覆盖）:
  POST /api/test/batch      → 由 suite + benchmark 探针 + 多配置覆盖
  POST /api/test/benchmark  → 由 suite + benchmark 探针覆盖
  POST /api/test/burn       → 由 suite + benchmark 探针 + 大量轮次覆盖
```

### 旧端点如何处理

暂不删除旧端点，前端逐步迁移到新端点后标记为 deprecated。

## 四、前端变化

### 新增"测试套件"标签页

替代当前的"批量测试 + 基准测试 + 压力测试"三个独立标签页：

```
配置管理 | 连接测试 | 聊天测试 | 🆕 测试套件 | 日志
```

**测试套件标签页布局：**
```
┌─ 选择配置 ─────────────────────────────┐
│  ☑ OpenAI (gpt-4o-mini)                │
│  ☑ DeepSeek (deepseek-chat)            │
│  ☐ Ollama (llama3)                     │
└────────────────────────────────────────┘
┌─ 选择探针 ─────────────────────────────┐
│  ☑ 基准测试 (并发/延迟/P50/P95)        │
│  ☑ JSON 模式                           │
│  ☐ 多轮对话                            │
│  ☐ 行为验证                            │
│  ☐ 温度稳定性                          │
└────────────────────────────────────────┘
┌─ 参数 ─────────────────────────────────┐
│  测试消息: [____________________]       │
│  轮次: [10]  并发数: [5]               │
│  [▶ 运行测试套件]                       │
└────────────────────────────────────────┘
┌─ 结果 = 配置 × 探针 ──────────────────┐
│       基准测试        JSON模式      多轮│
│ OpenAI  89分/120ms   95分/100%      —   │

```

**核心交互：** 选择配置 → 选择探针 → 一键运行 → 得到对比矩阵

## 六、与旧方案对比

| 维度 | 补丁式方案（旧） | Probe 框架（新） |
|------|---------------|----------------|
| 新增一个探针 | 新建 handler + route + 前端 tab | 实现 Probe 接口 + 注册 |
| 测试类型扩展 | 每加一个类型就要重复构造请求/解析响应 | 所有探针复用 Provider.Chat() |
| 多配置对比 | 各 handler 自己实现 | Suite 自动管理 |
| 统一报告 | 无，各 handler 输出不同格式 | 统一 Result 结构 |
| 前端复杂度 | 每个 tab 独立实现结果展示 | 一个对比矩阵组件展示所有 |
| 可维护性 | 4 个 handler 各自为政 | 1 个 suite handler + N 个 Probe |
| 风险 | 零（保留旧端点） | 低（旧端点可用，逐步迁移） |

## 七、需要确认的问题

1. **旧标签页保留还是移除？** 建议保留但标记为"经典模式"，新用户默认用测试套件
2. **`probe/` 包独立还是放在 `llm/` 下？** 建议独立，避免循环依赖
3. **探针注册方式？** 简单工厂模式 `probe.Registry["benchmark"] = &BenchmarkProbe{}`
