# llm-tester 项目开发规则

## 核心铁律（必须遵守）

### 1. Git 提交规范
- 每次代码变更务必提交到 Git 仓库
- 所有 commit message 统一使用中文

### 2. 文档同步
- 代码修改必须同步更新 docs/ 目录下的对应文档
- 文档命名：`序号_中文文件名_日期.md`
- 采坑记录模板：背景 → 原因 → 排查过程 → 解决方案 → 建议

### 3. 架构原则
- **小而美**：不做大而全的系统，做瑞士军刀，不过度设计
- **全局视野**：每次改动前必须先理解全局，不能打补丁
- **闭环思维**：所有功能必须形成完整闭环，并有单元测试覆盖

### 4. 技术选型
- **不重复造轮子**：能用行业标准库/框架就用，优先复用参考项目代码
- **ORM > 原始 SQL**：必须使用 ORM 防止 SQL 注入
- **Go 标准**：遵循 Go 官方最佳实践和代码规范

### 5. 代码质量
- 核心代码必须有详细的中文注释（变量、常量、业务流程）
- 单元测试必须写正式脚本，不得写了删删了写
- 不要在不必要的细节上耗费时间，集中精力实现核心功能

### 6. 引用项目
- 参考项目：`/Users/dingyuwang/0-X/0-AI工具/CLIProxyAPI/llm_auth_manager/cmd/token-refresher-gui`
- 复用模块：`concurrent/worker.go`、`errors/errors.go` 直接复制修改
- 复用架构：`main.go` 的 Gin + embed 模式、Vue 3 SPA 前端模式
