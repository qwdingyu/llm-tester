## 坑001: go.sum 未提交导致 CI 编译失败

- **背景**: 首次配置 GitHub Actions 自动构建
- **时间**: 2026-07-01
- **影响范围**: GitHub Actions CI 工作流
- **严重程度**: 🔴 阻塞

### 问题现象

CI 中 `go vet` 报错：
```
missing go.sum entry for module providing package github.com/gin-gonic/gin
```

### 根因分析

`go.sum` 文件在本地存在，但从未提交到 git 仓库。Go 1.23+ 在 CI 环境中需要 `go.sum` 验证依赖完整性。没有 `go.sum` 时，`go vet` 拒绝执行。

### 排查过程

1. 首次 CI 运行失败，查看日志发现 `missing go.sum entry`
2. 检查本地 `git status`，发现 `go.sum` 在 untracked 列表
3. 本地 `go vet` 正常，因为本地已有 `go.sum`

### 解决方案

```bash
git add go.sum
git commit -m "添加 go.sum"
git push
```

### 预防建议

- 初始化 Go 项目后立即提交 `go.sum`
- 在 `.gitignore` 中不要忽略 `go.sum`

## 坑002: go.mod 缺少 indirect 依赖

- **背景**: 定位并修复坑001后仍 CI 失败
- **时间**: 2026-07-01
- **影响范围**: GitHub Actions CI 工作流
- **严重程度**: 🔴 阻塞

### 问题现象

CI 中 `go vet` 报错：
```
updates to go.mod needed; to update it: go mod tidy
```

### 根因分析

`go.mod` 只列出了直接依赖（`github.com/gin-gonic/gin v1.10.0`），没有列出间接依赖。Go 1.23+ 工具链需要完整的依赖树做依赖完整性校验。

初始的 `go.mod`：
```
module github.com/user/llm-tester
go 1.23
require github.com/gin-gonic/gin v1.10.0
```

缺少 20+ 个间接依赖（如 `golang.org/x/net`、`google.golang.org/protobuf` 等）。

### 解决方案

```bash
go mod tidy
git add go.mod go.sum
git commit -m "补充 indirect 依赖"
```

`go mod tidy` 自动扫描代码中所有 import，补充完整的间接依赖树。

### 预防建议

- 每次新增依赖后执行 `go mod tidy`
- `go.mod` 和 `go.sum` 应一起提交
- 本地开发后 CI 失败时，先检查 `go.mod` 是否完整

## 坑004: 连接测试路径 /models 应为 /v1/models

- **背景**: 用户使用 unity2 API 测试连接失败
- **时间**: 2026-07-01
- **影响范围**: 所有 OpenAI 兼容 API 的连接测试
- **严重程度**: 🟠 功能缺陷

### 问题现象

连接测试返回"连接失败: 请求失败"，但聊天端点正常工作。

用户日志：
```
14:32:52 ❌ [unity2] 连接失败: 请求失败
14:33:14 💬 [unity2] 失败: 网络错误: 请检查网络连接后重试
```

### 根因分析

`OpenAIProvider.TestConnection()` 使用 `baseURL + "/models"` 路径。
- 当 `base_url = "https://api.unity2.ai"`（不含 `/v1`）时：
  - 连接测试: `GET https://api.unity2.ai/models` → HTTP 404 ❌
  - 聊天测试: `POST https://api.unity2.ai/v1/chat/completions` → HTTP 200 ✅

### 排查过程

1. 用 curl 分别测试 `/models` 和 `/v1/models`：
   ```bash
   curl https://api.unity2.ai/models → HTTP 404
   curl https://api.unity2.ai/v1/models → HTTP 200
   ```
2. 确认聊天端点正常：`POST /v1/chat/completions` 返回 200 并有回复内容
3. 确认预设模板中的 `base_url` 也包含 `/v1`，导致双重路径问题

### 解决方案

1. 将连接测试路径从 `/models` 改为 `/v1/models`
2. 将预设模板（OpenAI/DeepSeek/Kimi）的 base_url 去掉末尾的 `/v1`
3. 保留特殊 API 的原始路径（GLM-4、DashScope、Azure）

### 预防建议

- 所有端点路径应保持一致的版本前缀（`/v1`）
- 预设模板的 URL 不应包含 API 版本前缀
- 特殊 API（GLM-4、DashScope）需要单独验证 URL 格式

## 坑005: POST 请求缺少 stream=false 导致 60s 超时

- **背景**: 用户反馈聊天测试和批量测试全部超时
- **时间**: 2026-07-01
- **影响范围**: 所有 Provider 的 POST 请求
- **严重程度**: 🔴 阻塞

### 问题现象

聊天测试等待 60 秒后失败：
```
60000ms · 0 tokens
网络错误: 请检查网络连接后重试
```

连接测试正常（GET 请求），但所有 POST 请求（聊天、批量、基准、Burn）全部超时。

### 根因分析

大量 OpenAI 兼容 API 在未指定 `stream` 参数时，默认返回 **SSE 流式响应**（Server-Sent Events）。Go HTTP 客户端的 `io.ReadAll(resp.Body)` 会持续读取流式响应，但由于流永远不会结束（持续推送事件），`io.ReadAll` 会一直阻塞直到超时（默认 60s）。

```
API 行为:
  不传 stream → 默认 stream=true → SSE 流式响应 → io.ReadAll 永远读不完
  传 stream=false → JSON 响应 → io.ReadAll 正常完成
```

### 排查过程

1. 用 curl 测试 API 端点：
   ```bash
   # 不传 stream 参数
   curl -X POST ... -d '{...}' → 正常返回（curl 自动处理 chunked 响应）
   
   # 但 Go 的 io.ReadAll 会一直读取直到连接关闭
   ```
2. curl 正常是因为 curl 读取 HTTP 响应时会按 chunked 编码逐块读取，遇到新的 data 事件就输出。
3. Go 的 `io.ReadAll` 则一直读取直到 EOF，而 SSE 流直到超时才关闭连接。

### 解决方案

在所有 POST 请求体中显式添加 `"stream": false`。

```go
// 通用请求体构建函数（所有 Provider 共用）
func BuildChatBody(model, message string, temperature float64, maxTokens int) map[string]interface{} {
    return map[string]interface{}{
        "model":       model,
        "messages":    []map[string]string{{"role": "user", "content": message}},
        "temperature": temperature,
        "max_tokens":  maxTokens,
        "stream":      false,  // 强制非流式响应
    }
}
```

### 预防建议

- 所有 LLM API 的 POST 请求必须显式指定 `"stream": false`
- 在 Provider 抽象层统一处理，避免每个 Provider 单独修补
- 使用共享的 `BuildChatBody()` 函数，而非每个 Provider 手写 body

## 坑008: API Key 文件权限过松（安全风险）

- **背景**: 安全审计
- **时间**: 2026-07-01
- **影响范围**: 所有用户
- **严重程度**: 🔴 安全

### 问题现象

`~/.llm_tester/configs.json` 中明文存储了所有配置的 API Key，但：
- 目录权限: `0755`（其他用户可读）
- 文件权限: `0644`（其他用户可读）

### 解决方案

```go
// 目录：仅所有者可读写执行
os.MkdirAll(configDir, 0700)

// 文件：仅所有者可读写
os.WriteFile(tmpPath, data, 0600)
```

### 预防建议

- 存储敏感信息的配置文件和目录必须使用 `0700`/`0600` 权限
- 所有新项目的文件权限默认从最小权限开始

## 坑009: WorkerPool goroutine 泄漏

- **背景**: 代码审查时发现并发实现的安全问题
- **时间**: 2026-07-01
- **影响范围**: 批量测试、基准测试
- **严重程度**: 🟠 资源泄漏

### 问题现象

1. `RunBenchmark` 中，当超时触发后所有 worker 退出，但主循环继续向 `pool.input` 写数据 → 永久阻塞
2. Worker 在 `p.output <- result` 处没有 `ctx.Done()` 保护，超时后阻塞泄漏

### 根因分析

```go
// 问题代码
for i := 0; i < totalReqs; i++ {
    pool.input <- i  // 如果所有 worker 已退出，永久阻塞
}
```

### 解决方案

```go
for i := 0; i < totalReqs; i++ {
    select {
    case pool.input <- i:
    case <-ctx.Done():
        pool.Shutdown()
        return pool.CollectResults()
    }
}
```

并在 worker 的 output 写入处也添加 `select` + `ctx.Done()` 保护。

### 预防建议

- 所有 channel 操作都应有超时或取消保护
- 并发代码必须考虑所有 goroutine 退出路径

## 坑010: Request Context 未传播到 WorkerPool

- **背景**: 代码审查
- **时间**: 2026-07-01
- **影响范围**: 批量测试
- **严重程度**: 🟠 资源浪费

### 问题现象

`RunBenchmark` 使用 `context.Background()` 创建超时上下文，忽略传入的 HTTP 请求 Context。当客户端断开连接（关闭浏览器、网络中断），服务器不会停止正在进行的批量测试。

### 解决方案

```go
// 传播请求 Context
func RunBenchmark(ctx context.Context, concurrency int, totalReqs int,
    timeout time.Duration, fn func(reqID int) TestResult) []TestResult {
    benchCtx, cancel := context.WithTimeout(ctx, timeout)
    defer cancel()
    // ...
}
```

### 预防建议

- 所有长时间运行的操作都应接受 `context.Context` 参数
- 使用 `context.WithTimeout(parentCtx, timeout)` 而不是 `context.WithTimeout(background, timeout)`

## 坑011: Benchmark/Burn 不检测客户端断开

- **背景**: 代码审查
- **时间**: 2026-07-01
- **影响范围**: 基准测试、压力测试
- **严重程度**: 🟡 资源浪费

### 问题现象

Benchmark 和 Burn 测试的循环中不检查客户端是否断开连接。如果用户关闭浏览器，后续轮次仍然发起 HTTP 请求。

### 解决方案

```go
ctx := c.Request.Context()
for i := 0; i < req.Rounds; i++ {
    if err := ctx.Err(); err != nil {
        addLog("客户端断开，提前终止")
        break
    }
    // ... 执行测试
}
```

### 预防建议

- Gin handler 中长时间运行的循环应定期检查 `c.Request.Context().Err()`
- 所有可能耗时超过 1 秒的操作都应考虑客户端断开场景

## 坑012: serveIndex 忽略 embed 错误

- **背景**: 代码审查
- **时间**: 2026-07-01
- **影响范围**: 前端加载
- **严重程度**: 🟡 错误处理

### 问题现象

```go
serveIndex := func(c *gin.Context) {
    data, _ := staticFS.ReadFile("static/index.html")  // 错误被忽略
    c.Data(http.StatusOK, "text/html; charset=utf-8", data)
}
```

如果 embed 失败（构建时缺少文件），`data` 为 nil，前端收到空响应，但日志无任何提示。

### 解决方案

```go
serveIndex := func(c *gin.Context) {
    data, err := staticFS.ReadFile("static/index.html")
    if err != nil {
        addLog("❌ 读取前端静态文件失败: %v", err)
        c.String(http.StatusInternalServerError, "前端页面加载失败")
        return
    }
    c.Data(http.StatusOK, "text/html; charset=utf-8", data)
}
```

### 预防建议

- `embed.FS.ReadFile` 的返回值必须检查错误
- 所有可能失败的静态资源加载都应有错误处理

## 坑013: 预设模板 URL 含有重复 /v1

- **背景**: 用户使用预设后连接测试失败
- **时间**: 2026-07-01
- **影响范围**: 预设模板
- **严重程度**: 🟠 功能缺陷

### 问题现象

预设模板的 `base_url` 包含 `/v1`（如 `https://api.openai.com/v1`），但代码中的 `buildEndpointPath()` 会再次添加 `/v1`，导致 `https://api.openai.com/v1/v1/chat/completions`。

### 根因分析

`buildEndpointPath()` 假设 `base_url` 是 API 的根地址（不含版本路径），始终返回 `/v1/chat/completions`。预设模板中的 `/v1` 是多余的。

### 解决方案

移除 OpenAI、DeepSeek、Kimi 预设模板中的 `/v1` 后缀。
保留 GLM-4（`/api/paas/v4` 是 API 固有路径）、DashScope（`/compatible-mode/v1` 是兼容模式路径）、Azure（使用独立路径格式）。

### 预防建议

- 预设模板的 URL 不应包含 API 版本前缀
- 所有端点路径由代码统一添加版本前缀
- 特殊 API 需要单独验证和标注
