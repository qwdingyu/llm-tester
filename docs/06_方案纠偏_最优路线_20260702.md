# 方案纠偏报告：什么才是最优路线

> 日期: 2026-07-02
> 背景: 在分析 CyberTAR 后提出的初始方案不理想，本文件进行自我纠偏
> 分析角色: 行业顾问 + 产品经理 + 资深用户

---

## 一、初始方案的 3 个判断错误

### ❌ 错误1：将"扩展厂商预设"列为 P0

**预设只是字符串。** 预设模板本质上就是 `Name + BaseURL + Model` 三组字符串。用户手动填写同样能达到效果。将 URL 列表从 8 个扩展到 15 个不改变 LLM Tester 的核心能力。

**正确的判断：** P0 应是真正影响用户体验的核心能力。

### ❌ 错误2：忽略了已有功能的质量问题

基准测试首轮经常失败：

```
data: {"req_id":0,"status":500,"error":"网络错误: 请检查网络连接后重试"}
data: {"req_id":1,"status":200,"content_preview":"Hi! How can I help?",...}
```

首轮失败的 `latency_ms` 被计入 P50/P95 统计，拉高了整个延迟指标。**在核心功能不稳定时堆砌新功能，是产品经理的常见错误。**

### ❌ 错误3：低估了"批量对比"的价值

批量测试是 LLM Tester 的**独特竞争优势**——CyberTAR 是浏览器单线程做不了并发对比，OpenAI 的 Playground 也只测单个模型。但当前批量测试只是简单的并行输出，没有真正的对比分析（差异高亮、胜出者标记、百分比差异）。

**差异化竞争的黄金机会没有抓住。**

## 二、纠偏后的最优方案

```
原则:
  1. 先修核心，再拓新功能
  2. 差异化竞争，不做"我有别人也有"的无意义堆砌
  3. 每次改动必须可验证、可测试
```

### 🔴 P0：修核心（必须先做）

| 问题 | 现状 | 修复方案 | 预估工作量 |
|------|------|---------|:---------:|
| Benchmark 首轮网络错误 | 每轮测试第 1 个请求必失败 | 预热机制：正式测试前发 1 次"预热"请求，结果不计入统计 | 30 分钟 |
| Benchmark 失败请求污染统计 | 500 的 latency 被计入 P50/P95 | handleBenchmark 中跳过 Status != 200 的请求 | 10 分钟 |
| 批量测试无对比分析 | 单一结果表，无对比 | 增加 vs 列：当前配置 vs 最佳（最快），标记胜出者 | 2 小时 |
| Benchmark 缺少质量指标 | 只有 P50/P95 速度指标 | 增加 finish_reason 比率、内容长度分布、成功率 | 1 小时 |

### 🟠 P1：新能力（差异化竞争）

| 功能 | 为什么是 P1 | 预估工作量 |
|------|------------|:---------:|
| json_mode 探针 | 生产中最常用的 API 功能，当前完全无法测试 | 1 小时 |
| multi_turn 对话测试 | 聊天测试的最基础功能缺失 | 1 小时 |
| 基准测试报告增加"准确率" | 在速度之外增加质量维度，差异化 benchmark | 1 小时 |

### 🟡 P2：生态（可后补）

| 功能 | 理由 | 预估工作量 |
|------|------|:---------:|
| 扩展厂商预设 | 纯字符串，低价值 | 30 分钟 |
| behavior_exact 探针 | 有 json_mode 后，此探针的独立性下降 | 1 小时 |
| temperature_zero 探针 | 价值有限，用户感知弱 | 30 分钟 |

### ⚪ 不会做的（偏离产品定位）

| 功能 | 理由 |
|------|------|
| 模型指纹识别 | 需维护庞大知识库，偏离"桌面测试工具"定位 |
| 社区报告系统 | 需后端 + 数据库 + 认证 |
| 安全探针全景 | 需要 NLP 知识库，超出 CLI 工具范围 |
| Anthropic 原生协议 | 可通过 OpenAI 兼容模式调用 |

## 三、最优方案的核心原则

### 为什么"先修核心"比"新增探针"更重要？

**用户视角：**
- 用户运行基准测试 → 首轮失败 → 对结果可信度产生怀疑
- 用户做批量对比 → 只能手动比较延迟数值 → 体验差
- 这些是**每天都会遇到的问题**

**行业视角：**
- LLM 测试领域的成熟工具（如 LangSmith、MLFlow）强调的是**可复现的对比分析**
- 单纯的"并发测速"是初级阶段，**带质量指标的对比测试**才是行业标准

### 为什么"差异化竞争"比"补齐探针"更重要？

**竞品分析：**
- CyberTAR 有 19 个探针，但它是浏览器单线程的
- OpenAI Playground 只能测单个模型，没有批量对比
- LLM Tester 的 WorkerPool 可以并发测多个配置，这是独一无二的优势

**正确策略：**
- 不要试图在"探针数量"上追赶 CyberTAR（追不上）
- 要在"并发对比分析"上做到极致（别人做不了）

## 四、实施路线图

### 阶段 1（2 小时）：核心修复

```
1. Benchmark 预热机制 + 跳过失败请求的统计
2. 确认 RunBenchmark 的结果完整性
3. benchmark 报告增加质量指标
```

### 阶段 2（3 小时）：差异化竞争

```
1. 批量测试增加对比分析（vs 最佳列 + 胜出者标记）
2. json_mode 探针
3. multi_turn 对话测试
```

### 阶段 3（1 小时）：生态

```
1. 扩展厂商预设
2. 增加 behavior_exact 探针
```

## 五、总结

| 版本 | 之前方案 | 纠偏后方案 |
|------|---------|-----------|
| P0 | 扩展厂商预设 + 探针 | **修复基准测试首轮失败 + 批量测试对比分析** |
| P1 | 多轮/multi_turn | **json_mode + multi_turn + 质量指标** |
| 核心思想 | "补齐 CyberTAR 有的" | **"把 LLM Tester 擅长的做到极致"** |

**一句话：不要在别人的赛道上追赶，要在自己的赛道上拉大差距。**
