# LLM Tester 发布与构建流程

## 一、项目概述

LLM Tester 是一个 Go 语言编写的 LLM API 测试工具，使用 Gin + Vue 3 SPA 架构。

GitHub: https://github.com/qwdingyu/llm-tester

## 二、构建方式

### 2.1 快速构建（当前平台）

```bash
# macOS / Linux
./build.sh

# Windows
.\build.ps1

# 通用方式
go build ./cmd/llm-tester/
```

### 2.2 跨平台构建（Makefile）

```bash
# 构建所有平台二进制到 dist/
make build-all

# 构建并打包为压缩包
make release

# 清理构建产物
make clean
```

构建产物位于 `dist/` 目录，支持 5 个平台：

| 平台 | GOOS | GOARCH | 文件名后缀 |
|------|------|--------|-----------|
| macOS Intel | darwin | amd64 | `-darwin-amd64` |
| macOS Apple Silicon | darwin | arm64 | `-darwin-arm64` |
| Linux x86_64 | linux | amd64 | `-linux-amd64` |
| Linux ARM64 | linux | arm64 | `-linux-arm64` |
| Windows x86_64 | windows | amd64 | `-windows-amd64.exe` |

### 2.3 启动

```bash
go build ./cmd/llm-tester/ && ./llm-tester
# → http://localhost:8912
```

端口可通过环境变量 `PORT` 修改。

## 三、GitHub Actions 自动构建

### 3.1 工作流文件

`.github/workflows/build.yml` 包含两个 Job：

**check** — push/PR 到 main 时触发：
1. `actions/checkout@v4` — 拉取代码
2. `actions/setup-go@v5` — 安装 Go（版本从 `go.mod` 自动读取）
3. `go vet ./...` — 静态检查
4. `go build ./cmd/llm-tester` — 编译验证

**release** — 推送 `v*` 标签时触发（依赖 check 通过）：
1. 拉取代码 + 安装 Go
2. `make release` — 全平台编译 + 打包
3. `softprops/action-gh-release@v2` — 创建 GitHub Release 并上传 artifacts

### 3.2 触发发布

```bash
git tag v1.0.0
git push origin v1.0.0
# Actions 自动构建并在 Release 页发布
```

### 3.3 手动发布

```bash
make release
gh release create v0.1.0 \
  --title "v0.1.0" \
  --notes "版本说明" \
  dist/*.tar.gz dist/*.zip
```

## 四、采坑记录

### 4.1 go.sum 未提交导致 CI 失败

**现象**: GitHub Actions 中 `go vet` 报错：
```
missing go.sum entry for module providing package github.com/gin-gonic/gin
```

**原因**: `go.sum` 文件虽然在本地存在，但被 `.gitignore` 忽略或从未提交到 git 仓库。Go 1.23 在 CI 环境中需要 `go.sum` 验证依赖完整性。

**解决**: 确保 `go.sum` 已提交：
```bash
git add go.sum
git commit -m "添加 go.sum"
```

### 4.2 go.mod 缺少 indirect 依赖

**现象**: CI 中 `go vet` 报错：
```
updates to go.mod needed; to update it: go mod tidy
```

**原因**: `go.mod` 只列出了直接依赖（`github.com/gin-gonic/gin`），没有列出间接依赖。Go 工具链需要完整的依赖树来做校验。

**解决**: 
```bash
go mod tidy    # 自动补充 indirect 依赖
git add go.mod go.sum
git commit -m "补充 indirect 依赖"
```

### 4.3 concurrent/stats.go 缺少 sort 导入

**现象**: CI 中 `go build` 报错：
```
concurrent/stats.go:39:3: undefined: sort
```

**原因**: `concurrent/stats.go` 使用了 `sort.Float64s()` 但 import 语句只写了 `"math"`，缺少 `"sort"`。该文件在本地有未提交的修改，但 git 仓库中的版本不完整。

**解决**: 提交完整的文件修改：
```bash
git add concurrent/stats.go
git commit -m "修复缺失的 sort 导入"
```

### 4.4 连接测试路径 /models 应为 /v1/models

**现象**: 连接测试返回"连接失败: 请求失败"，但聊天端点正常工作。

**原因**: `OpenAIProvider.TestConnection()` 使用 `baseURL + "/models"`，但 OpenAI 兼容 API 的正确路径是 `baseURL + "/v1/models"`。聊天端点 `/v1/chat/completions` 路径正确。

**分析**: 域名 `api.unity2.ai` 的 API 遵循 OpenAI 标准路由：

| 路径 | 用途 | HTTP状态 |
|------|------|---------|
| `GET /models` | ❌ 连接测试（旧代码） | 404 |
| `GET /v1/models` | ✅ 连接测试（修复后） | 200 |
| `POST /v1/chat/completions` | ✅ 聊天测试 | 200 |

**解决**: 将 `openai.go` 中的 `/models` 改为 `/v1/models`，与聊天端点保持一致。

**注意**: 部分 API 的 base_url 已包含版本路径（如 GLM-4 使用 `/api/paas/v4`，DashScope 使用 `/compatible-mode/v1`）。使用预设模板后需根据实际情况调整 base_url。

### 4.5 预设模板 base_url 不包含 /v1

**现象**: 预设模板的 base_url 包含 `/v1`（如 `https://api.openai.com/v1`），但代码中的 `buildEndpointPath()` 会再次添加 `/v1`，导致 URL 变成双重重叠。

**根因**: `buildEndpointPath()` 假设 base_url 是 API 的根地址（不含版本路径），返回 `/v1/chat/completions`。预设模板中的 `/v1` 是多余的，已在最新版本中移除。

**特殊 API**（base_url 保留原始路径）:
- GLM-4: `https://open.bigmodel.cn/api/paas/v4` — /api/paas/v4 是 API 固有路径
- DashScope: `https://dashscope.aliyuncs.com/compatible-mode/v1` — /compatible-mode/v1 是兼容模式路径
- Azure: `https://YOUR_RESOURCE.openai.azure.com` — Azure 使用独立路径格式

## 五、文件结构

```
llm-tester/
├── .github/workflows/build.yml    # GitHub Actions 构建工作流
├── Makefile                        # 跨平台构建（macOS/Linux）
├── build.sh                        # 快速构建脚本（macOS/Linux）
├── build.ps1                       # 快速构建脚本（Windows）
├── cmd/llm-tester/main.go          # 主入口 + Gin 路由 + API Handlers
├── llm/
│   ├── provider.go                 # Provider 接口 + Config + HTTP 客户端
│   ├── openai.go                   # OpenAI 兼容 API 实现
│   ├── ollama.go                   # Ollama 本地模型实现
│   └── custom.go                   # 自定义端点实现
├── concurrent/
│   └── worker.go                   # 并发工作池
├── storage/
│   └── store.go                    # 配置持久化（JSON文件）
├── errors/
│   └── errors.go                   # 结构化错误类型（当前未使用）
└── cmd/llm-tester/static/
    └── index.html                  # Vue 3 SPA 前端
```
